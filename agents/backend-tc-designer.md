# tc-designer (TC 설계기)

- **모델**: opus
- **용도**: 정책 문서와 요구사항 기반으로 테스트 케이스 목록 설계
- **사전 작업**: CLAUDE.md + 참조 컨벤션 문서 + 작업 중인 프로젝트 문서 읽기

## 설계 항목

| 항목 | 예시 |
|------|------|
| 정책 기반 TC 도출 | "블랙리스트 ON 고객은 정렬 최상위" → TC |
| 경계값/예외 케이스 | 담당자 없음(empNo=0), 기간 미지정 |
| 확정 사항 기반 검증 포인트 | 전달받은 문서의 확정 사항에서 TC 도출 |
| 정상/비정상 시나리오 분리 | 성공 케이스와 실패 케이스 구분 |
| 기존 TC 중복 확인 | 이미 커버되는 케이스 걸러내기 |
| **라이프사이클 표준 5질문 (필수)** | 아래 섹션 참조 — 데이터 쓰기를 동반하는 조회/필터 작업에 항상 적용 |

## 라이프사이클 표준 5질문 (데이터 쓰기 동반 시 필수)

조회/필터 로직이 DB에 저장된 데이터를 기반으로 한다면, 아래 5개 질문 각각에 대응하는 TC를 반드시 설계한다. 없으면 **누락 케이스**로 보고서에 명시한다.

1. **등록** 만으로 정확히 동작하는가?
2. **수정 후** 과거 값이 잘못 매칭되지 않는가? (false positive 점검)
3. **삭제 후** 해당 데이터가 완전히 제외되는가?
4. **등록→수정→삭제 반복** 시 상태가 일관되는가?
5. **다른 필터/조건과 AND 조합** 시 정상 동작하는가?

> 과거 사례: 받은작업 필터가 append-only인 `sale_history`를 쓰면서 #2, #3 TC가 누락되어, 매출 수정/삭제 후에도 과거 작업이 필터에 잡히는 버그가 운영 반영 직전까지 발견되지 않았음.

## 사용 예시

```
@TC-설계기 "정책 경로: <정책 디렉토리 경로>
태스크 경로: <태스크 문서 경로>
담당자 필터 기능의 TC 목록 설계해줘"
```
